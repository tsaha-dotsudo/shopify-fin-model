# Shopify Financial Model & Analytics Pipeline

A complete reporting system for a Shopify store, built for an owner who wants answers rather than dashboards. It pulls the handful of Shopify reports that actually drive decisions, keeps them as plain CSV files, and turns them into three things: a private Excel financial model with real costs, a public anonymized web dashboard, and a founder-facing PDF summary.

Every number and every sentence of commentary in all three outputs is generated by code from the source data. Nothing is typed by hand, so a refresh can never go stale or contradict itself.

```
Shopify (ShopifyQL)
        |
        v
   raw/*.csv  (5 files, gitignored, never leave your machine)
        |
        +--> build_model.py      --> <SHOP_NAME>_Financial_Model.xlsx   private
        +--> build_dashboard.py  --> dashboard.html                     public
        +--> build_deck.py       --> <SHOP_NAME>_Store_Analytics_Deck.pdf
```

## 1. Why this exists

Shopify's built-in analytics are good at counting things. They are poor at answering questions. Six specific gaps motivated this pipeline:

| Shopify gives you | This pipeline gives you |
|---|---|
| Metrics in isolation | Growth decomposed into visitors x conversion x order size, so you know *why* a month moved |
| Absolute numbers, private by necessity | Indexed and anonymized output, so performance can be shared publicly without revealing a rupee |
| Revenue, but no costs | An Excel model where your own cost assumptions produce true contribution margin |
| History, with no forward view | Current month at run-rate plus a three-month projection range |
| You must notice unusual months yourself | Anomalies flagged automatically at 1.5 standard deviations, with the maths shown |
| Charts you have to interpret | Every chart carries a computed what / how / watch / conclusion card in plain language |

## 2. The reports and why only these

Four monthly reports plus one optional weekly report. Everything downstream is derived from these; nothing else is collected.

| File | Columns | Answers |
|---|---|---|
| `raw/pnl_monthly.csv` | month, orders, gross_sales, discounts, returns, net_sales, shipping_charges, taxes, total_sales | Is the store growing, and is growth being bought with discounts? |
| `raw/products.csv` | product_title, orders, gross_sales, discounts, returns, net_sales | Where does the money come from, and how concentrated is it? |
| `raw/customers_monthly.csv` | month, customers, returning_customers, returning_customer_rate, average_order_value | Are buyers coming back, and are orders getting bigger? |
| `raw/funnel_monthly.csv` | month, sessions, cart_adds, reached_checkout, completed_checkout, conversion_rate | Is a revenue change a traffic problem or a conversion problem? |
| `raw/weekly.csv` *(optional)* | week, orders, net_sales, total_sales, sessions, conversion_rate | What happened in the last few weeks, operationally? |

### The queries

Run these in Shopify (Analytics -> ShopifyQL, or via the Admin API) and export each as CSV. Replace `<start-date>` with your store's first month.

```sql
-- raw/pnl_monthly.csv
FROM sales SHOW orders, gross_sales, discounts, returns, net_sales, shipping_charges, taxes, total_sales
TIMESERIES month SINCE <start-date> UNTIL today

-- raw/products.csv
FROM sales SHOW orders, gross_sales, discounts, returns, net_sales
GROUP BY product_title SINCE <start-date> UNTIL today ORDER BY net_sales DESC LIMIT 30

-- raw/customers_monthly.csv
FROM sales SHOW customers, returning_customers, returning_customer_rate, average_order_value
TIMESERIES month SINCE <start-date> UNTIL today

-- raw/funnel_monthly.csv
FROM sessions SHOW sessions, sessions_with_cart_additions, sessions_that_reached_checkout,
sessions_that_completed_checkout, conversion_rate TIMESERIES month SINCE <start-date> UNTIL today

-- raw/weekly.csv  (optional; merge these two on week)
FROM sales SHOW orders, net_sales, total_sales TIMESERIES week SINCE -8w UNTIL today
FROM sessions SHOW sessions, conversion_rate TIMESERIES week SINCE -8w UNTIL today
```

If `raw/weekly.csv` is absent, the dashboard builds without the weekly tab. Everything else is required.

## 3. Setup and running it

```bash
pip install pandas openpyxl reportlab
```

Set `SHOP_NAME` at the top of each script. Drop the CSVs into `raw/`. Then:

```bash
python build_model.py       # -> <SHOP_NAME>_Financial_Model.xlsx   (private, gitignored)
python build_dashboard.py   # -> dashboard.html                     (public)
python build_deck.py        # -> <SHOP_NAME>_Store_Analytics_Deck.pdf
```

No database, no server, no build step, no API keys. The scripts read CSVs and write files.

## 4. The Excel model, sheet by sheet

`build_model.py` produces a workbook. Blue text is an input you can change, yellow fill means fill it in, black is a formula, green pulls from another sheet.

- **README** — what the workbook is, the four queries, the colour legend.
- **Data_PnL / Data_Products / Data_Customers / Data_Funnel** — the raw reports, untouched, one sheet each. Do not edit; these are what every formula points at.
- **Assumptions** — five yellow input cells: COGS as a % of net sales, courier cost per order, packaging cost per order, payment gateway %, and monthly fixed costs. The shipped values are placeholders to show the format, not real costs.
- **Model** — the monthly view, entirely formulas. Rows: orders, gross sales, discounts, returns, net sales, shipping charged, total sales, MoM growth, discount rate, return rate, AOV, shipping per order, sessions, conversion, cart adds per session, checkout completion, revenue per session, customers, repeat rate, then estimated COGS, courier, packaging, gateway fees, fixed costs, contribution and contribution margin. Change one Assumptions cell and every contribution figure recalculates.
- **Charts** — trend charts for net sales, orders, sessions and conversion.
- **Insights** — commentary computed at build time: growth versus run-rate, traffic-versus-conversion attribution, shipping cost recovery, repeat rate, product concentration, discount and return discipline, AOV.

This workbook contains real revenue and product names. It is gitignored and should never be committed to a public repo.

## 5. The dashboard, tab by tab

`dashboard.html` is a single self-contained file: no backend, no build step, no data fetching at runtime. Everything is baked in at generation time.

**Start here** — six questions with a computed yes/no verdict pill, one plain sentence each, and a sparkline: Is the store growing? Are visitors buying? Do customers come back? Are discounts paying for growth? Is the store fragile? Is each visit worth more? Verdicts are threshold-driven in code, not written by hand.

**Overview** — KPI pills, the health scorecard (donut plus five component bars), and the net sales index with rolling average and projection cone.

**This week** *(only if `raw/weekly.csv` exists)* — four delta pills comparing the last three full weeks against the three before, plus a weekly index chart with the current partial week starred.

**Growth** — growth decomposition (stacked: visitors, conversion, order size), the momentum heatmap across five metrics, and auto-detected anomalies.

**Customers & funnel** — conversion and repeat rate with a fitted trend projected to month 12; the funnel as it stands now plus all three stage-rates across every month.

**Economics & products** — AOV index, the discount-versus-growth scatter, product concentration bars and the concentration meter.

**Deep dive** — four efficiency dials (orders index, sessions index, revenue per session index, shipping recovery %), cart abandonment versus checkout completion, and the quarterly rollup.

**Method** — the pipeline, the anonymization scheme, the maths, a glossary, and the integrity note.

Every chart is paired with a four-row explainer card: *what this shows*, *how we built it*, *what to look for*, and a full *conclusion* with a recommendation. The conclusion text is generated from the data, and some conclusions branch on the values themselves; the discount-correlation verdict, for example, picks between three different readings depending on the computed correlation.

## 6. Every metric, defined

| Term | Definition |
|---|---|
| **Gross sales** | Product revenue before discounts, returns, shipping and tax |
| **Discounts** | Value given away through discount codes and automatic discounts |
| **Returns** | Value refunded |
| **Net sales** | Gross sales minus discounts minus returns. The headline revenue figure used throughout |
| **Total sales** | Net sales plus shipping charged plus tax. What actually hit the account |
| **Orders** | Count of completed orders |
| **AOV** | Average order value, net sales divided by orders. The size of a typical order |
| **Sessions** | Visits to the store |
| **Conversion rate** | Buyers per 100 visitors |
| **Cart adds** | Sessions where something was added to the cart |
| **Reached checkout** | Sessions that got as far as the checkout page |
| **Completed checkout** | Sessions that ended in a purchase |
| **Cart abandonment** | 1 minus purchases divided by cart adds |
| **Checkout completion** | Purchases divided by sessions that reached checkout |
| **Returning customer rate** | Returning buyers per 100 buyers in that month |
| **Revenue per session** | Total sales divided by sessions. What an average visit is worth |
| **Shipping recovery** | Shipping charged as a % of net sales. Compare against real courier cost |
| **Index** | A value divided by its first-full-month value, times 100. Hides absolute amounts |
| **Run-rate** | A partial period scaled to full-period pace |
| **Contribution** | Net sales plus shipping charged, minus COGS, courier, packaging, gateway fees and fixed costs |
| **`*` suffix** | A partial period shown at run-rate |
| **`p` suffix** | A projected period |

## 7. Every calculation, shown

**Run-rate for a partial period**
`value / days_elapsed * days_in_period`. Applied to the current month and current week so a half-finished period is never compared against a full one.

**Index**
`value / first_full_period_value * 100`. Used for net sales, orders, sessions, AOV and revenue per session.

**Month-over-month growth**
`index[i] / index[i-1] - 1`.

**Growth decomposition**
Net sales is the product of three factors: `sessions x conversion x AOV`. Taking natural logs turns that product into a sum, so `ln(sessions_i / sessions_i-1)`, the same for conversion, and the same for AOV, are three additive contributions that reconstruct the month's growth. They are shown stacked, in percentage points.

**Rolling average**
Simple three-period mean of the index.

**Projection cone**
Take `ln(sales_i / sales_i-1)` for every full month, compute the mean `mu` and standard deviation `sd`. Then for k months ahead: midpoint `last * exp(mu*k)`, upper bound `last * exp((mu+sd)*k)`, lower bound `last * exp((mu-sd)*k)`. Compounding in log space is what keeps the cone's shape honest for a growing series.

**Repeat-rate trend**
Ordinary least squares fit through monthly repeat rate against month number, extended to month 12.

**Anomaly detection**
For each month's MoM growth, `z = (growth - mean_growth) / sd_growth`. Anything with `|z| >= 1.5` is flagged, and the flag text states the value, the direction and the z-score.

**Concentration index**
Herfindahl-style: sum of squared shares for the top five products, plus the pooled remainder's squared share spread over an assumed 25 products. Scored against 0.25, the threshold economists treat as high concentration, and rendered as a meter position from 0 to 100.

**Health score**
Five sub-scores clamped to 0-100, then averaged with equal weight:
- conversion vs a 3% benchmark
- repeat rate vs a 30% benchmark
- returns vs a 5% ceiling (inverted; lower is better)
- discounts vs an 8% ceiling (inverted)
- three-month average growth vs 15% monthly

**Discount efficiency**
Pearson correlation between monthly discount rate and monthly growth. The written conclusion branches: below 0.3 reads as no relationship, 0.3 to 0.6 as suggestive but inconclusive at this sample size, above 0.6 as a real link worth pricing consciously.

**Weekly 3-vs-3**
Sum the last three full weeks, sum the three before, take the percentage change. Conversion is compared as a session-weighted average rather than a mean of rates, so a big week doesn't get the same weight as a small one. The partial week is excluded from both sides.

## 8. The anonymization scheme

The dashboard is safe to publish because nothing absolute survives into the HTML source:

- Months become `M1`, `M2`, ... `Mn`; weeks become `W1`, `W2`, ...
- Net sales, orders, sessions, AOV and revenue per session are all indexed to their first full period = 100
- Products are relabelled `Product A` through `Product E`, with everything else pooled as `All others`
- Only rates, ratios, shares, indexes and correlations appear
- No rupee amount, order count, customer count or visitor count is written to the file
- The generated date is deliberately omitted so the masked months cannot be mapped back to real calendar months

The Excel model does the opposite by design: it holds the real figures and stays private.

## 9. How the code is organized

Both builders follow the same two-part shape.

**`build_dashboard.py`**
1. *Load and normalize* — read the CSVs, drop the launch stub month, compute run-rates and indexes
2. *Derive* — every series in section 7: decomposition, rolling mean, cone, fit, anomalies, HHI, health, funnel rates, weekly deltas
3. *Write the prose* — a `card(what, how, watch, conclusion)` helper builds each explainer; conclusion strings are f-strings over the computed values, with branching where the reading depends on the number
4. *Render* — one HTML template string with `%TOKEN%` placeholders, filled by a chain of `.replace()` calls. All series are serialized into a single `D` object consumed by the chart code

**`build_model.py`**
1. Load the CSVs
2. Write data sheets verbatim
3. Write the Assumptions sheet with yellow input cells
4. Write the Model sheet as *formula strings* pointing at the data sheets, never as computed values, so the workbook stays live
5. Add charts, then generate the Insights text
6. A recalculation pass verifies zero formula errors

**`build_deck.py`** draws the PDF directly with ReportLab: eight landscape pages, hand-placed cards and sparklines, same palette as the dashboard.

Adding a chart to the dashboard means: compute the series, add it to `D`, add a `%TOKEN%` and its markup, add a `card(...)` entry, add a Chart.js block, and add the `.replace()` call. Nothing else needs to change.

## 10. Design system

Neutral dark, editor-like, no colour cast. Background `#131313`, cards `#1A1A1A`, raised panels `#202020`, hairline borders `#2C2C2C`. Text runs `#EDEDED` / `#A3A3A3` / `#6F6F6F`. A single warm sand accent `#E0A458` carries the primary data, with muted sage `#9AA88F` for secondary series and desaturated green and amber for verdicts. Inter, weights 400/500/600 only. Cards at 16px radius, controls at 12px, chart bar caps at 6px. Motion: sections fade-slide in on scroll, numbers count up, bars sweep with `cubic-bezier(0.22, 1, 0.36, 1)`, cards lift 2px on hover, all disabled under `prefers-reduced-motion`. Charts strip their chrome: no in-canvas legends, no axis borders, faint gridlines, 2px lines with slight tension. On mobile the tab bar becomes a horizontally swipeable strip under a frosted blur, and every two-column row stacks.

## 11. Security

- Chart.js loads from CDN with a `sha384` subresource integrity hash and `crossorigin`, so a tampered copy will refuse to execute
- The embedded JSON escapes `</` so no value can break out of its `<script>` tag
- `.gitignore` blocks `raw/*.csv` and `*_Financial_Model.xlsx`
- **Manual uploads bypass gitignore.** The GitHub web uploader will accept anything you drag in. Never drag real CSVs or the workbook into a public repo
- Deleted files remain in git history; removing them properly requires rewriting history, so prevention is the only cheap fix
- The page requests Google Fonts, which exposes visitor IPs to Google. Self-host Inter if that matters to you

## 12. Deployment

**Vercel** (recommended). Sign in with GitHub, Add New -> Project, import the repo, framework preset Other, leave build command and output directory empty, Deploy. `vercel.json` rewrites `/` to `/dashboard.html` so the root URL works. Every push to `main` redeploys automatically within about a minute, and every past deployment stays available for one-click rollback.

**GitHub Pages** works too: Settings -> Pages -> Deploy from a branch -> main. The page then lives at `https://<user>.github.io/<repo>/dashboard.html`.

Either way the site serves `dashboard.html` as a static file; it does not run the Python. A data refresh only reaches the live page when the regenerated `dashboard.html` is committed.

---

## 13. Refresh workflow

1. Re-run the queries in section 2, export fresh CSVs into `raw/`
2. `python build_model.py` for the private workbook
3. `python build_dashboard.py` for the public page
4. `python build_deck.py` if you need an updated summary
5. Commit **only** `dashboard.html` (and the deck, if you want it in the repo)

All written conclusions, verdicts, anomaly flags and projections regenerate against the new data. Nothing needs editing by hand.

## 14. Troubleshooting

**Charts don't render on the live site.** Almost always the integrity hash: if cdnjs's copy of Chart.js ever differs from the pinned `sha384`, the browser blocks it silently. Check the browser console; the fix is to update the hash or drop the attribute.

**`FileNotFoundError: raw/...csv`.** Run the scripts from the repo root, not from inside `raw/`.

**Weekly tab missing.** `raw/weekly.csv` doesn't exist, or its columns don't match the header in section 2. This is intentional behaviour, not an error.

**A percentage looks impossible.** Check whether the period is partial. Partial months and weeks are run-rated, so early in a month the projection can swing hard; the `*` suffix marks these everywhere.

**Excel shows `#REF!` or blank contribution rows.** The Assumptions cells are empty. Fill B4 through B8.

**The deck's numbers disagree with the dashboard.** They are built from the same CSVs, so this means one was generated before a data refresh. Rerun both.
